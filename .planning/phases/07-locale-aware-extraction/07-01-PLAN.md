# Phase 07 Plan 07-01: Locale-aware recipe extraction (preserve the source content's language)

<context>
norish's recipe-extraction LLM path (the structured "captions/description/page-text -> title+description+ingredients+steps" step) produced an ENGLISH recipe from a Dutch-language source in a real import on the verify stack ("Broodje Hete Kip"): the `name` stayed Dutch (extracted ~verbatim) but the AI-generated/normalized free-text fields (`description`, `recipeIngredient`, `recipeInstructions`, `notes`, `keywords`) came out English.

ROOT CAUSE: the shared extraction prompt template `packages/shared-server/src/ai/prompts/recipe-extraction.txt` (byte-identical twin in `packages/api/src/ai/prompts/recipe-extraction.txt`) is written ENTIRELY IN ENGLISH and contains NO language directive. The three prompt builders in `packages/api/src/ai/prompts/builder.ts` (`buildRecipeExtractionPrompt`/HTML, `buildImageExtractionPrompt`/images, `buildVideoExtractionPrompt`/video+captions) all start from `loadPrompt("recipe-extraction")` and never thread any source-language or locale signal. The per-call `system` strings in the call sites (`packages/api/src/video/normalizer.ts`, `recipe-parser.ts`, `image-recipe-parser.ts`) are also English. With an English instruction surface and no "match the source language" rule, Gemini (gemini-2.5-flash) defaults to English for everything it GENERATES/normalizes, while the title survives because it is copied near-verbatim from the Dutch captions. For this nl-first deployment (DEFAULT_LOCALE=nl) the recipe must stay in the SOURCE content's language.

SIGNALS AVAILABLE at the call sites:
- `getDefaultLocale()` (exported from `@norish/config/server-config-loader`) returns the configured default locale code (here `nl`, env-default `en`). All three call sites already import from this module.
- `VideoMetadata.language` (`packages/api/src/video/types.ts`) — a BCP-47 code of the video's ORIGINAL audio (e.g. "en","es","nl"), already populated and passed to `downloadCaptions` in `youtube.ts`. `extractRecipeFromVideo` already receives the full `metadata`, so this per-source language is readable WITHOUT changing the youtube.ts call site.
- `LOCALE_CATALOG` (`packages/i18n/src/locales.ts`) maps locale codes -> human language names (nl -> "Nederlands"), synchronous, no DB.

FIX STRATEGY (smallest change that actually fixes the LIVE bug):
- Do NOT edit `recipe-extraction.txt`. At runtime `loadPrompt` reads the prompt from `getPrompts()` (DB/config, seeded from the .txt), so editing the seed would NOT change an already-seeded deployment (the verify stack). Threading the directive through the BUILDER code guarantees it applies on every extraction regardless of config state — more robust AND minimal.
- Add a new synchronous prompt fragment `buildLanguageInstruction(targetLanguage?)` (mirrors the existing `buildAllergyInstruction` fragment shape) that appends a clear directive: produce ALL free-text recipe fields in the SAME language as the source content; if a target language is known, name it explicitly; the ONLY exception is the fixed `categories` enum (Breakfast/Lunch/Dinner/Snack, which the normalizer matches against an English enum). Map a locale CODE to a human name via LOCALE_CATALOG with a graceful fallback to the raw code.
- Add `targetLanguage?: string` to `RecipeExtractionPromptOptions` (inherited by the video/image option types) and append the new fragment in all three builders.
- Thread a real locale at the three call sites: video prefers `metadata.language` (the true source language) and falls back to `getDefaultLocale()`; HTML and image use `getDefaultLocale()` (no per-source signal there). The "match the source content's language" clause is the ultimate guard for mixed-language sources.

SCOPE GUARD: no new config, no UI, no env var, no schema/migration change. The `categories` enum stays English on purpose. The fragment helper is pure/sync so the unit test stays hermetic (no DB/config).
</context>

<interfaces>
- packages/api/src/ai/prompts/fragments/language.ts (NEW):
  - `localeToLanguageName(localeOrLang?: string): string | undefined` — looks the code up in LOCALE_CATALOG (exact, then base subtag before "-"), returns the human name; if no match returns the raw trimmed input; returns undefined for empty/undefined.
  - `buildLanguageInstruction(targetLanguage?: string): string` — returns a "\nLANGUAGE:" fragment. Always instructs: write name/description/notes/ingredients/steps/keywords in the SAME language as the source content; do NOT translate; keep the `categories` values in English (Breakfast/Lunch/Dinner/Snack). When `targetLanguage` resolves to a name, adds: "The source content is in {Name}; produce all of the above fields in {Name}." (kept as the explicit target + the source-match guard for safety).
- packages/api/src/ai/prompts/fragments/index.ts: re-export the two new symbols.
- packages/api/src/ai/prompts/builder.ts:
  - `RecipeExtractionPromptOptions` gains `targetLanguage?: string`.
  - `buildRecipeExtractionPrompt`, `buildImageExtractionPrompt`, `buildVideoExtractionPrompt` each append `buildLanguageInstruction(options.targetLanguage)` into their assembled prompt (next to the existing allergy/auto-tagging fragments).
  - `buildImageExtractionPrompt(allergies?, targetLanguage?)` gains the optional 2nd arg (image path has no options object today).
- packages/api/src/ai/prompts/index.ts: re-export `buildLanguageInstruction`, `localeToLanguageName`.
- packages/api/src/video/normalizer.ts: import `getDefaultLocale`; compute `const targetLanguage = metadata.language || (await getDefaultLocale())` and pass `targetLanguage` into `buildVideoExtractionPrompt`.
- packages/api/src/ai/recipe-parser.ts: import `getDefaultLocale`; pass `targetLanguage: await getDefaultLocale()` into `buildRecipeExtractionPrompt`.
- packages/api/src/ai/image-recipe-parser.ts: import `getDefaultLocale`; pass `await getDefaultLocale()` as the 2nd arg to `buildImageExtractionPrompt`.
- packages/api/__tests__/ai/prompts/builder.test.ts: add a `buildLanguageInstruction` describe block (pure, no mocks) asserting the directive text + the explicit-language line + the categories-English exception + the locale-code->name mapping + the no-arg fallback to source-match-only.

No env var. No new dependency. No schema change.
</interfaces>
