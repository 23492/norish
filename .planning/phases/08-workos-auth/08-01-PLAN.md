# Phase 08 Plan 08-01: WorkOS AuthKit as an additional better-auth login provider

<context>
Kiran is pivoting norish toward a possible commercial product and wants WorkOS AuthKit built in as an ADDITIONAL login provider, keeping better-auth as the session/user/household core. Additive + reversible: do NOT touch the existing email/password, Google, GitHub, or generic-OIDC providers.

AUTH WIRING (investigated):
- `packages/auth/src/auth.ts` builds better-auth with the **`genericOAuth`** plugin (better-auth/plugins) — `plugins: [ genericOAuth({ config: buildOIDCProviders() }), ... ]`. The existing OIDC slot is a genericOAuth provider (`providerId:"oidc"`, `discoveryUrl`, custom `getUserInfo`). There is NO `sso` plugin. Social providers (google/github) go through `socialProviders` (better-auth native).
- The client (`packages/shared/src/lib/auth/client.ts`) already loads `genericOAuthClient()`, so `signIn.oauth2({ providerId })` works. The login button dispatch (`apps/web/app/(auth)/login/components/provider-button.tsx`) sends github/google → `signIn.social`, EVERYTHING ELSE → `signIn.oauth2({ providerId })`. So a `workos` provider needs NO button-dispatch change — it falls into the oauth2 branch like oidc.
- Provider config is stored in `server_config` (DB), admin-editable at runtime via the tRPC admin router (`packages/trpc/src/routers/admin/auth-providers.ts`) + the admin UI (`apps/web/app/(app)/settings/admin/components/auth-providers/`). Secrets (`clientSecret`, `apiKey`) are encrypted/masked by `server-config.ts` repo (SENSITIVE_FIELDS already includes both). Providers are loaded into an in-memory cache at startup (`packages/api/src/startup/seed-config.ts::loadAuthProvidersIntoCache`) which `auth.ts` reads via `provider-cache.ts` getters. Auth-provider changes are RESTART_REQUIRED.
- The signup→own-cookbook hook is `databaseHooks.user.create.after` in auth.ts: it fires for ALL signups regardless of provider (it keys on user creation + `getHouseholdsForUser(user.id).length === 0`, idempotent) — so WorkOS-provisioned users get their own cookbook for free. The OIDC claim-processor runs only `if (account.providerId === "oidc")`, so WorkOS will NOT trigger OIDC claim processing (correct).

WORKOS SPECIFICS (verified against workos.com/docs + better-auth docs, NOT guessed):
- WorkOS AuthKit / User Management is a non-standard OAuth2 API at `api.workos.com` (NOT a standard OIDC IdP with a usable discovery doc for this flow). So it does NOT reuse norish's OIDC discovery slot.
- Authorization endpoint: `GET https://api.workos.com/user_management/authorize` — required `client_id`, `redirect_uri`, `response_type=code`; optional `provider=authkit` (hosted AuthKit UI), `state`, PKCE params, `screen_hint`.
- Token exchange: `POST https://api.workos.com/user_management/authenticate` with `client_id`, `client_secret` (= the WorkOS API Key), `grant_type=authorization_code`, `code`. Response returns the `user` object DIRECTLY (`{ object,id,first_name,last_name,email,email_verified,profile_picture_url,... }`) plus `access_token`/`refresh_token`/`organization_id`. There is NO standard `/userinfo` endpoint — the profile comes back in the token response.
- Credentials: Client ID (`client_...`) + API Key (`sk_...`). The API Key is used as the OAuth `client_secret`.

CHOSEN MECHANISM: a better-auth `genericOAuth` provider `providerId:"workos"` with EXPLICIT `authorizationUrl` + a custom `getToken` (POST authenticate, stash the full response in `raw`) + a custom `getUserInfo(tokens)` that reads `tokens.raw.user` and maps WorkOS profile → `{ id,email,emailVerified,name,image }`. This is the documented better-auth path for providers whose profile is in the token response (docs: "Implement Custom Token Exchange and User Info Fetching for Generic OAuth").

CALLBACK URI (must be reported to Kiran for the WorkOS dashboard): better-auth genericOAuth mounts `${baseURL}/oauth2/callback/:providerId`; norish's full mount is `${AUTH_URL}/api/auth/oauth2/callback/workos` (AUTH_URL = SERVER_CONFIG.AUTH_URL).

SCOPE GUARD: config is admin-runtime ONLY (NO env var, NO WorkOS env-seeding in seed-config — unlike OIDC/Google which have env import; WorkOS is intentionally admin-only per Kiran). No hardcoded keys. WorkOS reuses the GENERIC `AuthProviderForm` (2 fields: Client ID + API Key) like google/github — NOT a custom form. The two byte-identical server-config zod twins (`packages/config/src/zod/server-config.ts` + `packages/shared/src/contracts/zod/server-config.ts`) MUST get the identical edit. i18n keys for the admin form go in ALL 11 locales (nl+en real, 9 EN-fallback) — the login button text reuses the existing parameterized `auth.provider.signInWith` key (no new login i18n).
</context>
