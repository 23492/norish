# Phase 09 Plan 09-01: WorkOS auth provider configured via env (config-as-code) + admin-UI card removed

<context>
Phase 08 shipped WorkOS AuthKit as a better-auth `genericOAuth` provider whose config (Client ID + API Key) was admin-runtime ONLY — entered in Settings → Admin → Auth Providers, stored in the `auth_provider_workos` server-config row, cached at boot via `loadAuthProvidersIntoCache`, and consumed by `buildWorkOSProviders()` in `packages/auth/src/auth.ts`. Kiran (lead/owner) now wants WorkOS configured purely via ENVIRONMENT VARIABLES (config-as-code, set in the backend/compose at deploy time), never via the admin UI.

norish ALREADY has an env→provider mechanism for OIDC/Google/GitHub: `packages/api/src/startup/seed-config.ts::importEnvAuthProvidersIfMissing()` calls `syncOIDCProvider()/syncGitHubProvider()/syncGoogleProvider()`, each of which reads its env vars from `SERVER_CONFIG` (`packages/config/src/env-config-server.ts`) and SEEDS THE DB at boot (insert if no row; update a non-overridden row when env differs; delete a non-overridden row when env is removed; never touch an `isOverridden=true` row). After seeding, `loadAuthProvidersIntoCache()` reads the DB rows into the in-memory provider cache that `auth.ts` consumes. So the pattern is ENV SEEDS THE DB AT BOOT, then DB → cache → better-auth.

WorkOS was conspicuously the ONLY provider with no `syncWorkOSProvider()` and no `WORKOS_*` env vars — it could only be set via the UI. This plan mirrors the Google env pattern for WorkOS and removes the WorkOS admin-UI card.

SCOPE: change only the CONFIG SOURCE. Keep `buildWorkOSProviders()` (providerId `workos`, the `api.workos.com/user_management` authorize+authenticate flow, `authorizationUrlParams:{provider:'authkit'}`, custom getToken/getUserInfo), the `auth_provider_workos` server-config key + `AuthProviderWorkOSSchema` (clientId+apiKey), `getCachedWorkOSProvider`, provider-cache loading, and the tRPC router/connection-test/context plumbing all intact. Do NOT touch OIDC/Google/GitHub/email-password. Do NOT hardcode any real key. Do NOT `pnpm docker:build`, merge to main, deploy, or touch live containers.
</context>

<interfaces>
- packages/config/src/env-config-server.ts (ServerConfigSchema): add `WORKOS_CLIENT_ID: z.string().optional()` + `WORKOS_API_KEY: z.string().optional()` after the GOOGLE vars.
- packages/api/src/startup/seed-config.ts:
  - `hasOAuthEnvConfigured()` += `(SERVER_CONFIG.WORKOS_CLIENT_ID && SERVER_CONFIG.WORKOS_API_KEY)` so password-auth defaults off when only WorkOS env is set.
  - `importEnvAuthProvidersIfMissing()` += `await syncWorkOSProvider();`.
  - NEW `syncWorkOSProvider()` mirroring `syncGoogleProvider()`: hasEnvConfig = WORKOS_CLIENT_ID && WORKOS_API_KEY; envConfig = {clientId, apiKey, isOverridden:false}; insert if no row; if isOverridden skip; else update when configsDiffer; if no env and row exists non-overridden → deleteConfig. setConfig sensitive=true (API key encrypted). (`AuthProviderWorkOS` type already imported.)
- apps/web/app/(app)/settings/admin/components/auth-providers/auth-providers-card.tsx: REMOVE the `key="workos"` AccordionItem, the `tWorkos` hook, the `authProviderWorkOS` destructure, and `workos:false` from dirtySections. google/github/oidc + the generic AuthProviderForm untouched.

- packages/api/__tests__/startup/auth-provider-sync.test.ts: NEW "WorkOS Provider Sync (env-based config)" describe — insert-from-env, env-over-DB precedence, no-rewrite-when-equal, isOverridden respected, clientId-only-not-seeded, delete-when-removed.

NO new dependency. NO schema/migration (server_config is a generic KV table). The genericOAuth provider logic + the auth_provider_workos zod schema/cache/tRPC plumbing are unchanged. NO live-container/boot-patch touch. NO docker:build.
</interfaces>
