# Phase 11 Plan 11-01: Fix BROKEN WorkOS login — switch genericOAuth to standard OIDC via AuthKit discovery

<context>
WorkOS login is BROKEN on production. `POST /api/auth/sign-in/oauth2` with `{"providerId":"workos"}` returns 400 `{"code":"INVALID_OAUTH_CONFIGURATION"}`. better-auth rejects the WorkOS genericOAuth provider config at SIGN-IN.

ROOT CAUSE (confirmed in the installed better-auth@1.6.9 source): the `/sign-in/oauth2` endpoint handler (`node_modules/better-auth/dist/plugins/generic-oauth/routes.mjs`, the `signInWithOAuth2` endpoint) destructures `{ discoveryUrl, authorizationUrl, tokenUrl, ... }` from the matched provider config and resolves:
```js
let finalAuthUrl = authorizationUrl;
let finalTokenUrl = tokenUrl;
if (discoveryUrl) { /* fetch; finalAuthUrl = authorization_endpoint; finalTokenUrl = token_endpoint */ }
if (!finalAuthUrl || !finalTokenUrl) throw APIError.from("BAD_REQUEST", GENERIC_OAUTH_ERROR_CODES.INVALID_OAUTH_CONFIGURATION);
```
The endpoint requires BOTH an auth URL AND a token URL UP FRONT, before any token exchange. Crucially it does NOT consult `getToken` here — `getToken` is only used later in the callback's `validateAuthorizationCode`. The phase-08 WorkOS config (`buildWorkOSProviders()` in `packages/auth/src/auth.ts`) had `authorizationUrl` (so `finalAuthUrl` was set) but NO `tokenUrl` and NO `discoveryUrl` — so `finalTokenUrl` is `undefined` → INVALID_OAUTH_CONFIGURATION at sign-in. The custom `getToken` (POST api.workos.com/user_management/authenticate) could never run.

FIX (preferred, per the task): switch WorkOS to a STANDARD OIDC genericOAuth provider via AuthKit's OIDC discovery document. WorkOS AuthKit exposes a valid discovery doc at `https://<authkit-domain>/.well-known/openid-configuration` (CONFIRMED reachable, HTTP 200, from LXC 110 for `manageable-invention-37-staging.authkit.app`): issuer = the authkit domain, authorization_endpoint `.../oauth2/authorize`, token_endpoint `.../oauth2/token`, userinfo_endpoint `.../oauth2/userinfo`, jwks_uri `.../oauth2/jwks`, `token_endpoint_auth_methods_supported: ["none","client_secret_basic","client_secret_post"]`. better-auth resolves BOTH finalAuthUrl + finalTokenUrl from discovery → fixes the 400. Token exchange uses `client_secret_post` (better-auth's default in `validateAuthorizationCode`: it sets `client_id` + `client_secret` in the body when `clientSecret` is present). The WorkOS API Key (`sk_...`) IS the OAuth client_secret for AuthKit's authorization_code exchange (CONFIRMED via the WorkOS API reference — the authenticate cURL shows `"client_secret": "sk_example_..."`). OIDC claims (sub→id, email, name, picture→image) are mapped natively by better-auth's default `getUserInfo`, so the custom `getToken`/`getUserInfo` are dropped.

NEW per-environment, NON-SECRET env var `WORKOS_AUTHKIT_DOMAIN` (e.g. `manageable-invention-37-staging.authkit.app`) provides the discovery URL. The provider activates only when `WORKOS_CLIENT_ID` + `WORKOS_API_KEY` + `WORKOS_AUTHKIT_DOMAIN` are ALL set. providerId stays `workos`, so the callback stays `/api/auth/oauth2/callback/workos` (already registered in WorkOS). Keep the phase-09 env-seed mechanism (env→DB→cache), the admin-card removal (phase 09), and the SSO auto-redirect (phase 10).

SCOPE: reconfigure the `workos` genericOAuth provider + add one env var + thread it through the zod schema/env-seed. Do NOT touch OIDC/Google/GitHub/email-password. Do NOT hardcode any key. Do NOT `pnpm docker:build`, merge to main, deploy, or touch live/`/opt/norish/`.
</context>
