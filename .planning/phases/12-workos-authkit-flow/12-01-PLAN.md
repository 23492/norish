# Phase 12 Plan 12-01: Fix BROKEN WorkOS login — first-party AuthKit flow (api.workos.com authorize + custom getToken/authenticate), reverting the phase-11 OIDC-discovery surface

<context>
WorkOS login is STILL broken on production after phase 11. The phase-11 fix (this is the 3rd iteration) switched the `workos` genericOAuth provider to `discoveryUrl: https://${WORKOS_AUTHKIT_DOMAIN}/.well-known/openid-configuration` (the `*.authkit.app` OIDC document). `POST /api/auth/sign-in/oauth2 {providerId:workos}` now returns 200 and redirects to `…authkit.app/oauth2/authorize?client_id=<env client_id>`, but WorkOS then shows **"Authorization Error — The application you are trying to authorize was not found."**

ROOT CAUSE of the phase-11 failure (confirmed via the WorkOS docs): the `authkit.app/oauth2/*` OIDC endpoints are WorkOS **Connect / OAuth Applications** (third-party apps), which require a *separately-registered OAuth Application* client_id. The environment Client ID (`client_01KV3ADWK7MYAD4FBM0N14RCCC`) is NOT a Connect OAuth-Application client_id — it is the first-party **Environment Client ID**. The correct first-party **AuthKit** authorize endpoint is `GET https://api.workos.com/user_management/authorize` with the Environment Client ID + `provider=authkit`, and the token exchange is `POST https://api.workos.com/user_management/authenticate` (which returns the user object directly — there is no OIDC `/userinfo`). This is the phase-08 surface, which was abandoned in phase 11 only because better-auth's `/sign-in/oauth2` rejected the phase-08 config with `INVALID_OAUTH_CONFIGURATION` for lacking a token URL.

STEP-1 SOURCE VERIFICATION (installed better-auth@1.6.9, `node_modules/better-auth/dist/plugins/generic-oauth/routes.mjs`) — the precondition for this fix:
(a) SIGN-IN validity (`signInWithOAuth2`, lines 62-78):
```js
const { discoveryUrl, authorizationUrl, tokenUrl, ... } = config;
let finalAuthUrl = authorizationUrl;
let finalTokenUrl = tokenUrl;
if (discoveryUrl) { /* fetch; overrides both */ }
if (!finalAuthUrl || !finalTokenUrl) throw APIError.from("BAD_REQUEST", GENERIC_OAUTH_ERROR_CODES.INVALID_OAUTH_CONFIGURATION); // line 78
```
=> providing BOTH `authorizationUrl` AND `tokenUrl` (no discoveryUrl) satisfies the sign-in check. `tokenUrl` is NEVER fetched at sign-in; it only has to be a truthy string. CONFIRMED: (a) = YES.
(b) CALLBACK precedence (`oAuth2Callback`, lines 178-201):
```js
if (providerConfig.getToken) tokens = await providerConfig.getToken({ code, redirectURI, codeVerifier }); // line 179
else {
  if (!finalTokenUrl) throw ... INVALID_OAUTH_CONFIG;                                                    // line 185
  ... tokens = await validateAuthorizationCode({ ..., tokenEndpoint: finalTokenUrl, ... });              // lines 187-200 (standard POST to tokenUrl)
}
```
and then line 208:
```js
const userInfo = providerConfig.getUserInfo ? await providerConfig.getUserInfo(tokens) : await getUserInfo(tokens, finalUserInfoUrl);
```
=> when `getToken` is set, the standard token POST to `tokenUrl` is in the `else` branch and is SKIPPED entirely; the custom `getToken` runs and the custom `getUserInfo(tokens)` consumes its result. CONFIRMED: (b) = YES — `getToken` takes precedence over `tokenUrl` in the callback, and `getUserInfo` is fed the `getToken` output.

CONCLUSION: Option A is valid. We can supply `tokenUrl` PURELY to satisfy better-auth's sign-in config-validation (line 78), while the actual token exchange runs through the custom `getToken` (the api.workos.com/user_management/authenticate POST), exactly as phase 08 intended — but now the sign-in 400 is gone because `tokenUrl` is present. The authorize URL goes back to `api.workos.com/user_management/authorize` (first-party AuthKit) instead of `authkit.app/oauth2/authorize` (Connect), fixing the "application not found" error.

SCOPE: reconfigure `buildWorkOSProviders()` to the first-party AuthKit flow (Option A); REVERT the phase-11 `discoveryUrl`/`WORKOS_AUTHKIT_DOMAIN`/`authkitDomain` surface (the wrong OIDC surface) across auth.ts + the zod schemas + env-config + seed-config; gate the provider on `clientId + apiKey` only. KEEP the phase-09 env→DB→cache seed mechanism, the phase-09 admin-card removal, and the phase-10 SSO auto-redirect. Do NOT touch OIDC/Google/GitHub/email-password. Do NOT hardcode any key.
