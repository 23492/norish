# Phase 14 Plan 14-01: AI provider + transcription configured via env (config-as-code) + fix env↔DB re-seed drift

<context>
norish is going commercial server-hosted SaaS. Operator-level config (AI/transcription keys) must live in `.env` as the SOURCE OF TRUTH so it survives a clean DB / relaunch with zero admin clicks. Phase 09 already did this for WorkOS auth (`syncWorkOSProvider`): env → `auth_provider_workos` DB row, re-seeded on EVERY boot, env wins. R1 extends that pattern to the AI provider and transcription.

THE BUG (confirmed on live): after the clean-DB cutover, `server_config.ai_config` fell back to the DEFAULT `{provider:openai, model:gpt-5-mini, apiKey:empty}` → recipe import scrapes fine but AI extraction throws `PROVIDER_ERROR: "API Key is required for OpenAI provider"` (packages/shared-server/src/ai/providers/factory.ts). Root cause: `ai_config` (and `video_config`) are in `REQUIRED_CONFIGS` and only seeded via `seedMissingConfigs()`, which checks `configExists(key)` and SKIPS if a row exists. That is FIRST-BOOT-ONLY: a clean DB seeds from env once (and if `AI_API_KEY` is unset, it seeds an empty key), and any later admin-UI edit then DRIFTS — a stale DB row wins on every subsequent boot because there is NO `syncAIConfigFromEnv()` equivalent to `syncWorkOSProvider()`. The WorkOS quote of the gap: WorkOS has `syncWorkOSProvider()` (insert/update-on-diff/delete, runs every boot inside `importEnvAuthProvidersIfMissing()`); AI/video have NOTHING — `seedMissingConfigs()` is the only path and it is gated on `!configExists`.

A SECOND latent gap: the AI factory (factory.ts) ALREADY supports `deepseek`, `google`, `anthropic`, `mistral`, `perplexity`, `azure`, `groq` (phases 05/06), and `TranscriptionProviderSchema` already includes `assemblyai` — but the env `AI_PROVIDER` enum in env-config-server.ts is restricted to `["openai","ollama","lm-studio","generic-openai"]` and `TRANSCRIPTION_PROVIDER` omits `assemblyai`. So the lead's intended `AI_PROVIDER=deepseek` would FAIL zod env validation at boot today. R1 widens both env enums to match the canonical `@norish/config/zod/server-config` schemas (the single source of truth; `@norish/db/zodSchemas/server-config` re-exports it; the AI factory's `AIProvider` type is `AIConfig["provider"]`).

SCOPE: change only the CONFIG SOURCE + re-seed cadence for `ai_config`/`video_config`. Env vars `AI_*`/`TRANSCRIPTION_*`/`VIDEO_*` already exist in env-config-server.ts. Keep the admin-UI AI/transcription forms (that removal is R3). When env is unset, fall back to the current DB/admin behavior (no regression). Do NOT hardcode any real key. Do NOT `pnpm docker:build`, merge to main, deploy, or touch live / `/opt/norish/`.
</context>

<interfaces>
- packages/config/src/env-config-server.ts (ServerConfigSchema):
  - WIDEN `AI_PROVIDER` enum to the full set `["openai","ollama","lm-studio","generic-openai","perplexity","azure","mistral","anthropic","deepseek","google","groq"]` (mirror `AIProviderSchema`). Default stays `"openai"`.
  - WIDEN `TRANSCRIPTION_PROVIDER` enum to add `"assemblyai"` (mirror `TranscriptionProviderSchema`: openai/groq/azure/generic-openai/ollama/assemblyai/disabled). Default stays `"disabled"`.
  - No new env var NAMES are needed (AI_PROVIDER/AI_MODEL/AI_ENDPOINT/AI_API_KEY/AI_TEMPERATURE/AI_MAX_TOKENS/AI_TIMEOUT_MS/AI_ENABLED + TRANSCRIPTION_PROVIDER/TRANSCRIPTION_ENDPOINT/TRANSCRIPTION_API_KEY/TRANSCRIPTION_MODEL + VIDEO_PARSING_ENABLED/VIDEO_MAX_LENGTH_SECONDS/YT_DLP_* all already defined). norish uses a SINGLE `AI_API_KEY` (no per-provider keys) — keep that convention.
- packages/api/src/startup/seed-config.ts:
  - NEW `syncAIConfigFromEnv()` (mirrors syncWorkOSProvider; env wins, re-seeds EVERY boot): gate `hasEnvConfig = !!(AI_API_KEY || AI_ENDPOINT)` (covers cloud key + local endpoint). When set, read the existing row (`getConfig(AI_CONFIG, true)`), build the OPERATOR subset from env {enabled, provider, endpoint, model, apiKey, temperature, maxTokens, timeoutMs} and MERGE over the existing row so admin-behavior fields {visionModel, autoTagAllergies, alwaysUseAI, autoTaggingMode} are preserved; `setConfig(AI_CONFIG, merged, null, true)` (sensitive=true → apiKey encrypted) ONLY if `configsDiffer` from existing (no-op when unchanged). When env unset → do NOTHING (leave DB/admin row as-is; the REQUIRED_CONFIGS default still seeds an empty-key disabled row on a truly clean DB so reads never crash). AIConfig has NO isOverridden field, so env-when-set is unconditionally authoritative (simpler than WorkOS — exactly the requirement "don't let a stale DB row win").
  - NEW `syncVideoConfigFromEnv()` (transcription): gate `hasEnvConfig = TRANSCRIPTION_PROVIDER !== "disabled" && (TRANSCRIPTION_API_KEY || TRANSCRIPTION_ENDPOINT)`. Build the video/transcription subset from env and merge over the existing row (preserve nothing admin-only beyond what env owns; env owns transcriptionProvider/Endpoint/ApiKey/Model + enabled/maxLength/maxVideoFileSize/ytDlp*). `setConfig(VIDEO_CONFIG, merged, null, true)` only if changed.
  - WIRE both into `seedServerConfig()` AFTER `seedMissingConfigs()`/`normalizeExistingConfigs()` and after `importEnvAuthProvidersIfMissing()` (so the default row exists first, then env overwrites). Reuse the existing `configsDiffer` helper.
- .env.example + docker/docker-compose.example.yml: add an "AI & TRANSCRIPTION (operator config — env is source of truth)" block documenting AI_ENABLED/AI_PROVIDER/AI_MODEL/AI_API_KEY (+ AI_ENDPOINT for local) and VIDEO_PARSING_ENABLED/TRANSCRIPTION_PROVIDER/TRANSCRIPTION_API_KEY/TRANSCRIPTION_MODEL, noting env is re-seeded every boot and survives a clean DB.
- packages/api/__tests__/startup/ai-config-sync.test.ts: NEW suite mirroring auth-provider-sync.test.ts mocks — env→ai_config insert/update, env-over-stale-DB precedence (the drift fix), no-rewrite-when-equal, preserves admin-behavior fields, no-op when env unset, deepseek provider accepted; + a transcription/video env→video_config case.

NO new dependency. NO schema/migration (server_config is a generic KV table; AIConfig/VideoConfig schemas unchanged — providers already present). The AI factory, getAIConfig/getVideoConfig loaders, and admin AI/transcription forms are UNCHANGED. The contracts copy (packages/shared/src/contracts/zod/server-config.ts) already has all providers — not touched. NO live-container/boot-patch touch. NO docker:build.
