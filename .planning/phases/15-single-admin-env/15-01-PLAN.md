# Phase 15 Plan 15-01: env-driven server-admin allowlist (ADMIN_EMAILS) — first-user-owner replaced by an env allowlist

<context>
norish is going commercial server-hosted SaaS (one operator: Kiran). Today the SERVER admin/operator (who can reach `/settings/admin` — AI config, video/transcription, auth providers) is decided by a **"first user to sign up becomes server owner+admin"** rule baked into the better-auth user-create hook. That is wrong for SaaS: the first random person to register would own the server. R2 makes the server-admin set **env-authoritative** via a new `ADMIN_EMAILS` (comma-separated) var in `.env`, mirroring R1 (phase 14, `syncAIConfigFromEnv`): env is the single source of truth, re-evaluated on EVERY login, surviving a clean DB. Everyone NOT in `ADMIN_EMAILS` — INCLUDING the first signup — is a regular user.

CRITICAL distinction (do NOT conflate): this is the SERVER-LEVEL admin/operator flag `users.isServerOwner`/`users.isServerAdmin` (gated by `withServerAdmin`/`isUserServerAdmin` in trpc middleware, and the `admin.config.getUserRole` client tab gate). It is NOT the per-cookbook/household admin (`households.adminUserId`, POLICY-01 / `canAccessResource`) — that stays EXACTLY as-is; regular users still own + admin their OWN cookbooks/recipes/sharing. This phase touches ONLY the server-admin flags + the first-user rule. No household/cookbook ownership change.

CURRENT MODEL (what exists):
- `users.isServerOwner` + `users.isServerAdmin` (booleans, default false) in `packages/db/src/schema/auth.ts`. A partial unique index `user_single_server_owner_idx` enforces AT MOST ONE `isServerOwner = true`.
- The live "first user = owner+admin" lives in `packages/auth/src/auth.ts`, `betterAuth({ databaseHooks: { user: { create: { before } } } })`: `const isFirstUser = (await countUsers()) === 0; ... isServerOwner: isFirstUser, isServerAdmin: isFirstUser`. The `create.after` hook then disables registration after the first user. A SECOND, currently-unwired duplicate of the same rule lives in `createUser()` in `packages/db/src/repositories/users.ts` (exported, no production caller, no test) — it must be neutralized too for defense-in-depth.
- Server-admin is ONLY ever set at user-create today; there is NO per-login re-evaluation of the SERVER flag (only OIDC GROUP claims re-eval via `processClaimsForUser` → `setUserAdminStatus`). So existing users never get re-evaluated.
- Gates (UNCHANGED, both read the DB flags so they stay correct once the flags are env-authoritative): trpc `withServerAdmin` middleware → `isUserServerAdmin(userId)` (DB `isServerOwner || isServerAdmin`); trpc context → session `isServerOwner || isServerAdmin`; client settings tab → `admin.config.getUserRole` → `getUserServerRole` (DB). No gate code changes are needed.

EMAIL IS ENCRYPTED AT REST: `users.email` is AES-encrypted; `users.emailHmac` is a deterministic HMAC purely for lookup (`createEncryptedEmailAdapter` rewrites better-auth's `WHERE email=` to `WHERE emailHmac=`). PLAINTEXT email is available: (a) in the `user.create.before` hook as `user.email` BEFORE it is encrypted; (b) anywhere via `getUserById(userId)` which DECRYPTS (already used by `claim-processor.ts` + `withAuth.ts`). WorkOS login (live auth is WorkOS-only) populates `user.email` through `buildWorkOSProviders().getUserInfo` (`tokens.raw.user.email`), which better-auth feeds into the same create/update path → plaintext reaches the create hook. So the email match is done where plaintext is available: the create hook (`user.email`) for new users, and `getUserById` (decrypt) for the per-login re-eval.

SCOPE: add `ADMIN_EMAILS` env + a small pure allowlist module + wire it at user-create and at every login (session create). Replace the first-user-owner rule with: env allowlist authoritative WHEN SET; first-user-owner fallback ONLY when `ADMIN_EMAILS` is empty/unset (so existing self-host installs don't lock themselves out). Do NOT hardcode the email (real value `mailkiranknoppert@gmail.com` goes in `/opt/norish/.env` at deploy; tests use `admin@example.com`). Do NOT `pnpm docker:build`, merge to main, deploy, or touch live / `/opt/norish/`.
</context>

- packages/auth/src/admin-allowlist.ts (NEW — pure + testable, mirrors how claim-processor.ts is a separate module from the untestable betterAuth({...}) block):
  - `getAdminEmailSet(): Set<string>` — builds a lowercased/trimmed Set from `SERVER_CONFIG.ADMIN_EMAILS` (already normalized to string[] by the env transform; re-normalize defensively).
  - `isAdminEmail(email: string | null | undefined, adminSet: Set<string>): boolean` — case-insensitive + trim compare (email.trim().toLowerCase() ∈ set). Empty/missing email => false.
  - `isAllowlistConfigured(adminSet: Set<string>): boolean` — `adminSet.size > 0`.
  - `resolveServerAdminOnCreate({ email, isFirstUser, adminSet }): { isServerOwner: boolean; isServerAdmin: boolean }` — THE create-time decision:
      • allowlist CONFIGURED  => `{ isServerOwner: false, isServerAdmin: isAdminEmail(email, adminSet) }`  (env wins; first signup is NOT owner unless listed; never grant isServerOwner via env so the single-owner unique index can't be violated by a multi-email list — gate is `owner || admin` so admin alone fully enables `/settings/admin`).
      • allowlist EMPTY/UNSET => `{ isServerOwner: isFirstUser, isServerAdmin: isFirstUser }`  (PRESERVE current first-user-owner behavior — self-host fallback).
  - `applyServerAdminOnLogin(userId: string): Promise<void>` — the per-login re-eval (provider-agnostic). NO-OP when the allowlist is EMPTY/UNSET (don't churn self-host owners). When configured: `getUserById(userId)` to get PLAINTEXT email + current role via `getUserServerRole(userId)`; desired = `isAdminEmail(email)`; if `email ∈ list` and not already admin => `setUserAdminStatus(userId, true)`; if `email ∉ list` and currently owner-or-admin => DEMOTE via `setUserServerRole(userId, false, false)` (clears BOTH flags so a previously auto-owner who isn't the operator loses server-admin reach — requirement). Only writes on an actual change (no needless version bumps). Wrapped in try/catch + authLogger (a login must never hard-fail on this), mirroring claim-processor's after-hook error handling.
  - Imports `SERVER_CONFIG` from `@norish/config/env-config-server`, `{ getUserById, getUserServerRole, setUserAdminStatus, setUserServerRole }` from `@norish/db/repositories/users`, `authLogger` from `@norish/shared-server/logger` (same imports the sibling claim-processor.ts uses).
