# Phase 18 Plan 18-01: registration_enabled + password_auth_enabled via env (config-as-code) + fix env↔DB re-seed drift

<context>
norish is going commercial server-hosted SaaS. The auth toggles `registration_enabled` and `password_auth_enabled` must live in `.env` as the SOURCE OF TRUTH so they survive a clean DB / relaunch with zero admin clicks. Phase 09 did this for WorkOS auth (`syncWorkOSProvider`) and R1 (phase 14) extended the pattern to the AI provider + transcription (`importEnvOperatorConfig` → `syncAIConfigFromEnv`/`syncVideoConfigFromEnv`, env wins, re-seeded on EVERY boot). R4 extends the exact same pattern to the two auth toggles.

THE DRIFT (confirmed on the WorkOS-only launch): `registration_enabled` and `password_auth_enabled` are in `REQUIRED_CONFIGS` in `packages/api/src/startup/seed-config.ts` and were seeded ONLY through `seedMissingConfigs()`, which is gated `if (!await configExists(key))` — i.e. FIRST-BOOT-ONLY. So on a clean DB the toggles seed once from their hardcoded defaults (`registration_enabled => true`; `password_auth_enabled => !hasOAuthEnvConfigured()`), and env changes NEVER propagate afterward. Worse, `packages/auth/src/auth.ts` (user.create.after hook) sets `registration_enabled=false` after the FIRST user signs up — so for the commercial launch (registration must stay OPEN, password OFF since WorkOS-only) the operator had to run a MANUAL `UPDATE server_config ...` because there was no env→DB sync. There was no `syncAuthTogglesFromEnv()` equivalent to `syncWorkOSProvider()` / `syncAIConfigFromEnv()`.

A SECOND gap: the env layer had a DEAD `ENABLE_REGISTRATION` var (`packages/config/src/env-config-server.ts`, `.default(false)`) with NO consumer anywhere, and NO `PASSWORD_AUTH_ENABLED` env var at all. So neither toggle was reachable from env even in principle.

SCOPE: change only the CONFIG SOURCE + re-seed cadence for `registration_enabled`/`password_auth_enabled`. Keep the admin-UI toggles (trpc `admin/general` `updateRegistration`/`updatePasswordAuth`) — env simply re-seeds over them on the next boot when set (env is source of truth). When env is UNSET, fall back to the current DB/admin behavior with NO regression — incl. registration auto-disabling after the first user for self-host. Do NOT hardcode any value. Do NOT `pnpm docker:build`, merge to main, deploy, or touch live / `/opt/norish/`.
</context>

<interfaces>
- packages/config/src/env-config-server.ts (ServerConfigSchema):
  - REMOVE the dead `ENABLE_REGISTRATION` (unused; `.default(false)`).
  - ADD `REGISTRATION_ENABLED` + `PASSWORD_AUTH_ENABLED` as OPTIONAL booleans (`z.string().transform(v => v === "true" || v === "1").pipe(z.boolean()).optional()`). Optional (NO `.default()`) is load-bearing: it makes `SERVER_CONFIG.X === undefined` the "env unset" signal, so an explicit `false` (operator intent) is DISTINCT from unset (no-op). Mirrors how the AI sync gates on an optional `AI_API_KEY`.
- packages/api/src/startup/seed-config.ts:
  - NEW `syncAuthTogglesFromEnv()` (mirrors `syncAIConfigFromEnv`; env wins, re-seeds EVERY boot) + a small `syncBooleanToggleFromEnv(key, envValue, label)` helper. Per toggle: if `envValue === undefined` → NO-OP (leave DB/admin row). Else read the stored bool (`getConfig<boolean>(key)`) and `setConfig(key, envValue, null, false)` (non-sensitive plain bool) ONLY when `existing !== envValue` (no needless version bump). These keys have NO `isOverridden` field, so when the env var is SET env is UNCONDITIONALLY authoritative (a stale DB row — incl. the value `auth.ts` writes after the first user, or an admin-UI edit — never wins).
  - WIRE into `importEnvOperatorConfig()` after `syncAIConfigFromEnv()`/`syncVideoConfigFromEnv()`, so it runs on every boot AFTER `seedMissingConfigs()`/`normalizeExistingConfigs()` have ensured a default row exists.
- .env.example + docker/docker-compose.example.yml: add/replace the FIRST USER SETUP password block with an "AUTH TOGGLES (operator config — env is the SOURCE OF TRUTH)" note documenting REGISTRATION_ENABLED + PASSWORD_AUTH_ENABLED (re-seeded every boot, survives clean DB, unset = no-op) + the commercial WorkOS-only preset (REGISTRATION_ENABLED=true, PASSWORD_AUTH_ENABLED=false).
- packages/api/__tests__/startup/auth-toggles-sync.test.ts: NEW suite mirroring ai-config-sync.test.ts mocks — commercial preset overrides stale DB; env true beats the post-first-user auto-disable; env unset = no-op; no-rewrite-when-equal; env false (distinct from undefined) flips a DB true; seeds when no DB row; idempotent across two boots.

NO new dependency. NO schema/migration (server_config is a generic KV table; both toggle schemas already exist as `z.boolean()`). The readers `isRegistrationEnabled()` (@norish/config/server-config-loader) and `isPasswordAuthEnabled()` (@norish/auth/providers), the admin-UI toggles, and `auth.ts` are UNCHANGED — they read the same DB keys the new sync keeps in lockstep with env. NO live-container/boot-patch touch. NO docker:build.
</interfaces>
