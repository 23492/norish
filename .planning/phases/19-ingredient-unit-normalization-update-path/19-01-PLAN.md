# Phase 19 Plan 19-01: normalize locale-specific ingredient units on the recipe UPDATE path (parity with CREATE)

<context>
The recipe CREATE path normalizes locale-specific unit terms to canonical IDs, but the UPDATE path does NOT — so editing a recipe persists the un-normalized unit (e.g. Dutch "handvol" instead of "handful", "scheut" instead of "splash", "gr" instead of "gram"). This is the exact gap that `packages/db/__tests__/server/db/repositories/ingredient-unit-normalization.test.ts` flags: the 3 `updateRecipeWithRefs` tests fail while the `createRecipeWithRefs` tests pass. Phase 18's own SUMMARY/verification explicitly logged this as a PRE-EXISTING OUT-OF-SCOPE failure ("updateRecipeWithRefs / ingredient-unit-normalization (db, 3 failing)"); Phase 19 closes it.

ROOT CAUSE (confirmed by reading the source):
- CREATE: `attachIngredientsToRecipeByInputTx` in `packages/db/src/repositories/ingredients.ts` fetches the units config once via `getUnitsForNormalization()` (line ~199) and writes `unit: normalizeUnit(ri.unit ?? "", units)` for BOTH the existing-id rows (line ~216) and the newly-created-ingredient rows (line ~238).
- UPDATE: `syncRecipeIngredientsTx` in `packages/db/src/repositories/recipes.ts` (the function `updateRecipeWithRefs` delegates to) builds its row `values` with `unit: ingredient.unit ?? null` (line ~1285) — VERBATIM, no normalization. That is the only un-normalized write.

This fix was previously authored AND verified-by-eye as commit `17cb5659` ("fix(db): normalize ingredient units on the recipe update path") but was then RESET AWAY (git reflog: `reset moving to HEAD~1`) because it was a direct director hand-edit, which this fork forbids (delegate-via-gsd). Phase 19 re-applies the SAME minimal change through the sanctioned gsd flow (cross-AI worker → native Opus review), so it is verifiable and re-baseable.

SCOPE FENCE (do NOT widen):
- Touch ONLY the two files below. Do NOT change schemas, migrations, the create path, the readers, the trpc layer, or any test file.
- `recipes.ts:963` (`unit: ingredient.unit ?? null` inside the "saved copy" insert DTO builder) is NOT a second bug and is OUT OF SCOPE: that array is handed to `createRecipeWithRefs` → `attachIngredientsToRecipeByInputTx`, which normalizes it downstream. The ONLY un-normalized path is `syncRecipeIngredientsTx`.
- The normalization MUST mirror the create path EXACTLY: same helper (`getUnitsForNormalization`), same call shape (`normalizeUnit(ingredient.unit ?? "", units)` — empty-string fallback, NOT `?? null`), units fetched ONCE before the loop (not per-iteration). Do NOT invent a new helper or a different null-handling.
- No new dependency. No security surface: unit normalization canonicalizes a free-text string on the recipe being edited; it crosses no household/cookbook boundary, touches no auth/permission check, and changes no visibility logic (per-cookbook isolation is unaffected).
- Do NOT run `pnpm docker:build`, do NOT merge to main, do NOT deploy, do NOT touch live / `/opt/norish/`.
</context>
