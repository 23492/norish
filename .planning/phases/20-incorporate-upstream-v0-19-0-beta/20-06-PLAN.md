<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| beta env → live data | The beta stack must never read/write the live Postgres or share its encryption key; isolation is by separate DB/Redis/volume/network + a distinct MASTER_KEY. |
| public beta domain → WorkOS auth | A new public domain handling auth must have its callback registered so WorkOS sign-in is not hijackable/misrouted. |
| Cloudflare ingress → beta container | Public ingress routes anonymous traffic to the beta app; the route must point at the beta port, not live. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-06-DBLEAK | Information Disclosure | beta sharing/clobbering the live Postgres | mitigate | Separate `norish-beta-db` + volume `norish_beta_db_data` + dedicated network (compose); the clone script asserts the target URL contains `norish-beta` and refuses live; gate: `grep norish_db_data` returns zero in the beta compose (Task 1). |
| T-20-06-KEY | Information Disclosure / Cryptography | reusing live's MASTER_KEY on beta | mitigate | Beta env template carries a NEW placeholder MASTER_KEY with a comment that it MUST differ from live (live's encrypted secrets are key-incompatible); operator sets a fresh key in Task 2. |
| T-20-06-CB | Spoofing | WorkOS callback hijack on the new beta domain | mitigate | Register `https://norish-beta.knoppsmart.com/api/auth/oauth2/callback/workos` in the WorkOS dashboard before activating WorkOS on beta (Task 2 operator step). |
| T-20-06-LIVE | Tampering / Availability | accidental mutation of the live stack during beta deploy | mitigate | Beta is a NEW compose file + NEW services; the build is director-owned + detached; Task 2 is `autonomous:false` (operator), and the smoke-check confirms `norish-app`/`norish:live` unchanged. |
</threat_model>
