# Phase 20.1 — Replace @heroui-pro/react with free components — CONTEXT

**Requested by Kiran (2026-07-15):** "I do not want to pay for this key, so lets generate new ui components using the 21st dev mcp. Lets start with the main page recipe cards as a test."

## Why this phase exists

Phase 20 (incorporate upstream v0.19.0-beta) is code-complete on `integ/upstream-0.19.0`, but the deployability gate is BLOCKED: upstream 0.19.0 adopted **`@heroui-pro/react@1.0.0-beta.5`** — HeroUI **Pro**, a paid/token-gated library. Without `HEROUI_AUTH_TOKEN` (a paid license), `pnpm install` lands a stub package with no exports/css, so typecheck/vitest stay green (stub + test alias) but the real Next.js production build FAILS (`Can't resolve '@heroui-pro/react'`). See STATE.md 2026-06-28 entry.

**Decision (Kiran, 2026-07-15): do NOT buy the license. Replace every `@heroui-pro/react` usage with free equivalents** so the fork builds with zero paid dependencies, permanently. This also removes the CI secret dependency (`secrets.HEROUI_AUTH_TOKEN`) and the Dockerfile secret mount.

## Constraints

- Branch: `integ/upstream-0.19.0` (same branch as phase 20; this phase completes its build gate).
- Live stack (`norish-app`/`norish:live`) untouched; `pnpm docker:build` remains director-owned.
- Zero NEW npm dependencies: every replacement uses packages already in the tree (`@heroui/react` v3.1.0, `react-aria-components`, `embla-carousel-react` ^8.6.0). 21st.dev is used as the code/reference source, not as a package registry.
- Visual/behavioral parity is the acceptance bar for each swap: same placements, same interactions (swipe, snap-drag on sheets, dots, thumbnails, drag-drop + file picker), same a11y roles.
- gsd flow: 3 plans, serial. Verify per plan with typecheck + vitest; the phase-level gate is a clean `pnpm --filter @norish/web build` (Next.js production build — the exact step that failed) WITHOUT `HEROUI_AUTH_TOKEN`, then the director-owned `pnpm docker:build`.
