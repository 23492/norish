# Candidate Phase 22.3 — read-path isolation (VIEW-ISO-01 + MEDIA-AUTHZ-01)

**Written:** 2026-07-21, against `main` @ `73a34852`, live image `c5fb0e897946`.
**Status:** SCOPED, NOT PLANNED, NOT FIXED. Filed for a decision on sequencing.
**Severity:** VIEW-ISO-01 is a **live, active cross-cookbook read leak**. It is NOT closed
by Phase 22.2.

I went looking to scope MEDIA-AUTHZ-01 (filed yesterday as "real IDOR, not currently
exploitable, needs both UUIDs"). Establishing what access check the media routes *should*

## VIEW-ISO-01 — `canAccessResource("view", …)` honours a seeded per-cookbook `everyone`

### The defect

`packages/auth/src/permissions.ts:100-102`:

```ts
switch (policyLevel) {
  case "everyone":
    return true;
```

There is no cookbook-boundary clamp on that branch. The `household` branch below it is
carefully scoped (`requesterMemberHouseholdIds.includes(resourceHouseholdId)`, personal
recipes owner-only) — `everyone` bypasses all of it.

### Why that fires in production

Two facts combine, and each was verified rather than assumed:

1. **`view: "everyone"` is the shipped default.** `DEFAULT_RECIPE_PERMISSION_POLICY`
   (`packages/shared/src/contracts/zod/server-config.ts:381`) is
   `{view:"everyone", edit:"household", delete:"household"}`.
2. **Both live cookbooks carry `view_policy = 'everyone'` in the `households` table** —
   queried directly against the live DB on 2026-07-21:

   | id | name | view_policy |
   |---|---|---|
   | `0ca922db…` | Noa Van de Wetering's Cookbook | `everyone` |
   | `c44365cf…` | Kiran Knoppert's Cookbook | `everyone` |

`resolveRecipeCookbookPolicy` (`permissions.ts:52`) returns `cookbook.policy` **raw** — no
clamp — so `canAccessResource` receives `everyone` and returns `true` for any authenticated
requester.

### Verified, not inferred

Probed with live's exact values (temporary test, run and removed, never committed):

```ts
canAccessResource("view", "user-in-B", "user-in-A", "cookbook-A",
                  ["cookbook-B"], false,
                  {view:"everyone", edit:"household", delete:"household"}, "admin-of-A")
// ### cross-cookbook view allowed = true
```

And for personal recipes (`resourceHouseholdId = null`, policy resolved from the
server-wide default):

```ts
canAccessResource("view", "user-B", "user-A", null, [], false, SHIPPED_DEFAULT, null)
// ### canAccessResource(view, personal recipe of another user) = true
```

The second contradicts an invariant this repo states explicitly, in the header of
`packages/db/__tests__/server/db/repositories/households.isolation.test.ts`:

> …and that personal (household_id NULL) recipes stay **owner-only** while orphans
> (userId NULL) remain public.

### Reachable end-to-end

`packages/trpc/src/routers/recipes/recipes.ts:146` → `findRecipeForViewer`
(`routers/recipes/helpers.ts:84`) → `canAccessResource("view", …)`. So
**`recipes.getById` returns another cookbook's recipe to any authenticated user**, today,
on live.

`edit`/`delete` are NOT affected: they resolve to `household`, and that branch restricts to
the recipe's cookbook admin. **This is a read leak only.**

### Why Phase 22.2 did not close it, and why the suites stayed green

22.2 fixed `buildViewPolicyCondition` — the **list** query. `getById` is a different path
with a different gate. The system is now internally inconsistent: **the list hides a
recipe that the direct fetch will hand over.** That inconsistency is itself the argument
that one of the two is wrong, and the documented invariant says it is `canAccessResource`.

Note `buildViewPolicyCondition` already anticipated exactly this data shape
(`recipes.ts:365-370`): *"a freshly-created cookbook may still carry a seeded `everyone`
(inherited from a default-`everyone` global). So in the active-cookbook branch BOTH
`everyone` and `household` collapse to active-cookbook scoping"*. The **list layer clamps
the seeded `everyone`; the per-recipe gate honours it.** One of the two layers implemented
Decision #5 ("a per-cookbook `view = everyone` is disallowed") and the other did not.

This is the **fifth** instance of the family and the same test blind spot each time: the
trpc permissions suite is 278/278 green because it seeds `household`. Per the `AGENTS.md`
rule added yesterday — *if a test seeds a policy, it needs a sibling case on `everyone`* —
this was findable.

### Proposed fix

Clamp `everyone` to the cookbook boundary in `canAccessResource`, mirroring D-22-01's
"`everyone` stops at the cookbook boundary". Concretely: for `action === "view"`,
`everyone` should behave as `household` does — member-of-the-resource's-cookbook, and
owner-only when `resourceHouseholdId === null`. That is one edit at `permissions.ts:101`,
plus deciding whether truly-public reads (the `/share/[token]` path) still work — they
should, because that path does not go through `canAccessResource` at all.

**Caution:** this is the *permission kernel*. Every read in the app routes through it.
It needs the full failing-test-first + revert-check treatment, not a quick patch, and the
`/share/[token]` public path must be explicitly re-verified.

## MEDIA-AUTHZ-01 — restated now that its missing half is known

Yesterday's filing said: real IDOR, needs both recipe id and media filename, and
`RecipeDashboardSchema` omits media — so no single leak hands over the pair.

**That mitigation is weaker than stated.** `FullRecipeSchema` *does* carry `images` and
`videos`, and `getById` returns `FullRecipeSchema`. So VIEW-ISO-01 hands an attacker the
filenames for any recipe whose id they know, and MEDIA-AUTHZ-01 then serves the bytes with
no check at all. The two compose into a complete chain.

What still limits it: recipe ids must be known, and with Phases 22 / 22.1 / 22.2 landed,
the previous id-discovery channels (realtime broadcast, unfiltered list) are closed.

Unchanged from yesterday's filing:
- `app/(app)/recipes/[id]/[filename]/route.ts` and the `steps/` sibling: **0 hits** for
  `canAccess|getSession|household|auth`.
- `app/(app)/avatars/[id]/route.ts`: **0 hits** likewise.
- `lib/recipe-media.ts:127` does filename + path-traversal validation only.
- Sole gate is `apps/web/proxy.ts`, which checks **`session?.user` only**.

## Recommended sequencing

1. **VIEW-ISO-01 first.** It is live, it is the permission kernel, and fixing it shrinks
   MEDIA-AUTHZ-01 from "chain" back to "needs two UUIDs".
2. **MEDIA-AUTHZ-01 second**, reusing the (by then correct) gate.
3. Only then Phase 21.

**Both are ready to plan — neither needs a decision from Kiran.** The open question is
priority: this is a read leak between two accounts that belong to the same household in
practice, so real-world harm on *this* instance is low. But it ships default-on to every
deployment of the fork, and it is on the path Phase 3 was believed to have proven
adversarially.
