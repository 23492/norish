# 26-01 PLAN — Dinner suggester: scoped candidates + pure ranking (DINNER-01, backend)

**Requirement:** DINNER-01 (backend half)
**DB:** none — reads only existing tables; DB stays at migration 41.

## Context (grounded in code)
- `listRecipes` / `getRandomRecipeCandidates` (packages/db recipes repo) both scope via
  `buildViewPolicyCondition(ctx)` — the HOUSE-06 boundary. `getRandomRecipeCandidates`
  already pairs a scoped query with a pure `selectWeightedRandomRecipe` in shared-server:
  the exact pattern to mirror.
- `recipe_ratings` + tags are shipped surfaces. The DTO carries tags + avg rating; the
  candidate query additionally needs `max(updatedAt)` for recency.
- The rater name list is the ALREADY-gated `ratings.getRaters` (`assertRecipeAccess(view)`
  first) — reused by the UI, never re-implemented here.

## Decisions
- **D-26-01** candidate scoping inherited from `buildViewPolicyCondition` (never re-derived).
- **D-26-02** season derived from the recipe's OWN tags via a bilingual lexicon (no new source).
- **D-26-03** recency = household-scoped avg + count + `max(updatedAt)` last-rated.
- **D-26-04** ranking is pure + deterministic; date-seeded jitter, no `Math.random()`.

   Add the export-map entry in `packages/shared-server/package.json`.
