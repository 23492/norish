## Decisions taken FOR W2 (new; the executor implements them, does not relitigate)

These resolve genuine ambiguities in the §7 W2 row. Each is recorded so a verifier can see
it was a decision and not a drift.

### D-27-W2-01 — The `.cook` producer in W2 is the CALLER, never a runtime heuristic
`§7` says "new recipes write `cook_source` + projection", but the only thing that can turn
today's flat `steps[]` + `recipe_ingredients[]` into per-step Cooklang linkage is either
(a) the improved extraction prompt — **W3**, or (b) the heuristic name-matcher, which D-6/D-7
**ban from every runtime path** (it survives only as one-time migration glue in W5).

**Resolution:** W2 builds the write path and proves it through its own entry point; it does
NOT invent a producer. `createRecipeWithRefs` / `updateRecipeWithRefs` gain an **optional
extra argument** `cook?: { cookSource: string; cookTokens: CookTokensDTO }`. When supplied,
`cook_source` is stored and `deriveProjectionTx` runs. When omitted — which is *every*
call site that exists at the end of W2 — the legacy code path runs unchanged and
`cook_source` stays NULL. W3 flips the producers on; nothing else has to move.

### D-27-W2-02 — `cook` is a REPOSITORY argument, NOT a tRPC input field
`§2.8` says "create/update input accepts the per-step linkage shape". Accepting a raw `.cook`
string on `FullRecipeInsertSchema` would put **untrusted client text straight into the WASM
parser** (T-27-01) a whole wave before anything needs it.

**Resolution:** the tRPC input schemas are UNCHANGED this wave. The `cook` payload is an
extra repository parameter that only **server-side** producers (W3's extraction/import) can
fill. `grep` over `packages/trpc/src/routers/recipes/` must find no input schema carrying a
`.cook`. This is a must_have truth.

### D-27-W2-03 — The transitional read fork is delivered SERVER-SIDE in W2; the client fork is W4
`§7`'s "Renderer reads tokens when present, else old path" collides with W4, which owns the
token renderer. **Resolution:** W2 ships the *server* half — the DTO carries `cookTokens` when
`cook_source` parses and `null` otherwise. Web and mobile are **not touched**; they ignore the
new fields and keep rendering the legacy path. W4 adds the client-side
`cookTokens ? tokenRenderer : legacy` fork. No file under `apps/web/` or `apps/mobile/` may
appear in this plan's diff.

### D-27-W2-04 — `parseCookSource` returning `null` NEVER fails a user's write
W1 hands us: a `.cook` must parse with an EMPTY diagnostic report or `parseCookSource` returns
`null`. **Resolution, in both directions:**
- **Write:** `buildCookPayload` serializes then immediately parses its own output. If the parse
  returns `null`, `buildCookPayload` returns `null`, the caller passes **no** `cook` argument,
  `cook_source` is left/set to NULL, the **legacy projection write runs exactly as today**, and
  an **error-level** log (`recipeId`, step count, ingredient count, reason — **never the recipe
  text**) is emitted. The user's save SUCCEEDS. A failed derive must never surface as a 500.
- **Read:** a stored `cook_source` that fails to parse yields `cookTokens: null` (the client
  falls back to the legacy path) plus a warn-level log.
- **Invariant this buys us, and the reason it is the right call:** *a non-NULL `cook_source`
  always parses cleanly.* W4's renderer and W6's `0043` NOT-NULL both depend on that; a
  "store it anyway and hope" policy would silently poison them. The loud error log is what
  makes W5's backfill able to find the gap.

### D-27-W2-05 — Both systems, but only `recipe_ingredients` — opposite-system STEP PROSE is not synthesized
`§1.2` step 3 derives "other-system rows" with the deterministic converter. A converter can
convert an *amount*; it cannot rewrite step **prose** ("bake at 180 °C" → "bake at 350 °F").
**Resolution:** `deriveProjectionTx` materializes `recipe_ingredients` for **both** systems
(native verbatim; opposite via W0's `deriveConversion`, flag → preserve the native value) and
`steps` for the **native system only**. Opposite-system `steps` rows are left exactly as they
are. This is what makes the metric/US toggle safe today: see D-27-W2-06.

### D-27-W2-06 — The metric/US toggle short-circuit is tightened to require target-system STEPS
`recipes.convertMeasurements` short-circuits the AI conversion when
`recipe.recipeIngredients.some(ri => ri.systemUsed === targetSystem)`. Once
`deriveProjectionTx` materializes both systems' ingredient rows, that predicate becomes true
for a recipe that has **no target-system step prose**, and the toggle would show converted
ingredients next to un-converted steps. That is a user-visible regression.
**Resolution:** the short-circuit additionally requires at least one `steps` row in the target
system. Behaviour for every recipe that exists today is identical (a recipe converted by the
AI has both). The AI converter itself is **not** deleted — that is W6.

### D-27-W2-07 — The `0041` de-dup re-points grocery FKs FIRST, and flags what it merges
`groceries.recipe_ingredient_id` is `ON DELETE SET NULL`. A naive de-dup before the unique
index would silently null out shopping-list "from recipe X" links (the exact Phase 25 class of
defect). **Resolution, in this order, inside `0041`:** (1) pick the survivor per
`(recipe_id, system_used, ingredient_id)` = lowest `order`, tie-broken by lowest `id`;
(2) `UPDATE groceries SET recipe_ingredient_id = <survivor>` for every loser; (3) sum
`amount` into the survivor **only when every duplicate shares the same non-null `unit`, or all
units are NULL** — otherwise the survivor keeps its own amount/unit verbatim and the recipe is
marked `cook_review_needed = true` (the column lands in the same migration, and W5's review
queue is exactly where a lossy merge belongs); (4) delete the losers; (5) create the unique
index. Deleting a merged ingredient line is genuinely user-visible — see `<risks>` R1.

### D-27-W2-08 — Migrations in this fork are HAND-WRITTEN; do NOT run `pnpm db:generate`
`packages/db/src/migrations/meta/` holds snapshots only up to `0038`, while `0039` and `0040`
are hand-authored SQL. Running `drizzle-kit generate` would diff against the stale `0038`
snapshot and emit a duplicate/incorrect migration. **Resolution:** author
`0041_add_cook_source.sql` by hand, append the `_journal.json` entry (`idx: 41`,
`tag: "0041_add_cook_source"`, `version: "7"`, `breakpoints: true`, a fresh `when` epoch-ms),
and update `packages/db-schema` so the drizzle model matches the SQL. Do **not** add a
`0041_snapshot.json` (the fork stopped producing them at 0038 — do not restart the practice
mid-phase). The db test suite applies every `*.sql` in the folder in filename order via
testcontainers, so `0041` is executed by every db test run — that is the executor's proof.

### D-27-W2-09 — The WASM parser must never be reachable from `@norish/db`
`@norish/shared-server` → `@norish/db`. A parser import in the repository layer is a hard
cycle. **Resolution:** `deriveProjectionTx` takes an already-parsed `CookTokensDTO` — plain
data in, rows out. Parsing happens exactly twice, both in `@norish/shared-server`:
`buildCookPayload` (write) and `withCookTokens` (read). `packages/db/package.json` gains no
new dependency.

      where `DeriveProjectionReport = { ingredientRowsNative: number; ingredientRowsDerived: number; stepRows: number; flagged: Array<{ ingredient: string; reason: string }> }`.
      `tx` is the drizzle transaction handle the caller is already inside — this function NEVER
      opens its own transaction and NEVER touches the module-level `db`.
    - **Ingredient rows, native system:** one row per distinct ingredient across all steps, name

      the FIRST occurrence wins and the ingredient is reported in `flagged`
      (`reason: "mixed-units"`). `order` = first appearance order.
    - **Ingredient rows, opposite system:** for each native row, `deriveConversion` from
      `@norish/shared/units`. `ok` → converted quantity + unit. `flagged` → the row is written

      A density is NEVER invented (W0's guarantee, carried through).
    - **UPSERT stability (the keystone):** ingredient rows are written with
      `.onConflictDoUpdate({ target: [recipeIngredients.recipeId, recipeIngredients.systemUsed, recipeIngredients.ingredientId], set: { amount, unit, order, version: sql\`... + 1\`, updatedAt } })`.
      Rows are NEVER deleted-and-reinserted. Only ingredient rows for this recipe whose
      `ingredient_id` is absent from the new set are deleted, and only for the two systems this
      call wrote.
    - **Step rows, native system only** (D-27-W2-05): step prose is reconstructed from the token
      list as `tokens.map(t => t.type === "text" ? t.value : (t.name ?? "")).join("")`. A change
