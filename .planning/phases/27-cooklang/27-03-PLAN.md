**Phase 27 — WAVE 3 of 7: "Extraction native" (`27-ARCHITECTURE.md` §7, row W3).**

Make norish's AI extraction emit **per-step ingredient + timer linkage**, serialize that
linkage into a `.cook` with `buildCookPayload`, and hand it to the write path W2 already
built. After W3, a newly imported AI-extracted recipe is the first recipe in the database
with a non-NULL `recipes.cook_source`.

**Purpose.** W1 built the serializer and the parser read-model; W2 built the column, the
derived projection and the optional `cook` repository argument, and left every recipe with
`cook_source IS NULL` because nothing could produce per-step linkage. W3 is that producer.

**Output:** a linkage prompt fragment appended by all three extraction prompt builders; a
per-step `recipeInstructions` shape in `recipeExtractionSchema`; `buildCookFromExtraction`
in the extraction normalizer; an `ExtractedRecipe = { recipe, cook }` server-side channel
threaded from the three AI extractors through `ParseRecipeResult` into the three import
workers' `createRecipeWithRefs(..., cook)` calls; and — the security spine of this wave —
`@norish/shared-server/cooklang/limits`, the input-size cap enforced inside the two functions
that are the only doors to the WASM parser.

**W3 INHERITS T-27-01 (input-size limiting).** This is the wave where untrusted, model-shaped
text first becomes a `.cook` and reaches native code. Treat `@cooklang/cooklang` as a hostile-
input surface reached from user-influenced content: a scraped page, an uploaded photo or a
video transcript steers the model, and the model's output becomes the parser's input. Task 1
lands the caps and Task 5 proves them. No later task may open a path to the parser that
bypasses them.

**THE NEVER-BROKEN GUARANTEE, RESTATED AS A CONSTRAINT.** Every import that succeeds today
must still succeed, with the same ingredient rows, the same step rows and the same shopping-
list behaviour. `cook_source` is a bonus a recipe earns by passing two gates; failing either
gate is the ORDINARY case, not an error, and it costs the user nothing. Anything that makes an
import fail, lose an ingredient, or lose a step is a defect of this wave, not a feature.

**Hard scope fence — these belong to W4–W6 and MUST NOT be touched:** the web/mobile token
renderer, multi-timer, the `cookTokens ? tokenRenderer : legacy` client fork, deleting the
heuristic runtime path (W4); the `0042` backfill, `cook_confidence`, the confidence gate, the
review queue and the repair tool (W5); `0043` NOT-NULL, deleting `unit-converter.ts` /
`applyIngredientLinkMarkup` / `createIngredientLinkCandidates` / `SmartInstruction` / the
timer-keyword scan (W6). Decisions D-1..D-8, D-27-W1-01..06 and D-27-W2-01..09 are locked —
implement them, do not relitigate.

## Decisions taken FOR W3 (new; the executor implements them, does not relitigate)

The §7 W3 row is one sentence — "3 prompt builders + schema + serializer wiring; JSON-LD
importer → per-step linkage → `.cook`. New imports fully native + single-system extraction
(§4)." Each ambiguity in it is resolved below so a verifier can see a decision, not a drift.

### D-27-W3-01 — The linkage instruction is a CODE FRAGMENT, never an edit to `recipe-extraction.txt`
`loadPrompt("recipe-extraction")` resolves through `getPrompts()` — **server config**, seeded
once from the `.txt` and thereafter editable by an admin
(`packages/shared-server/src/ai/prompts/loader.ts`). Editing the `.txt` therefore changes
NOTHING on an existing install, including Kiran's. **Resolution:** the fragment is
`buildLinkageInstruction()` in `packages/api/src/ai/prompts/fragments/linkage.ts`, appended by
all three builders exactly like `buildAllergyInstruction` / `buildLanguageInstruction`. The
`.txt` files are NOT edited. This is a `must_haves` truth because the naive reading of §2.9
produces a silent no-op.

### D-27-W3-02 — The `.cook` travels as `ExtractedRecipe = { recipe, cook }`, a SERVER-SIDE pair
A `.cook` must never become a tRPC input (D-27-W2-01/02), and `FullRecipeInsertDTO` **is** the
tRPC input type (`recipes.create`, paste-import's `FullRecipeInsertSchema.safeParse`). Hanging
a `cook` key on that DTO — even optionally — would put it one `.extend()` away from being
client-supplied. **Resolution:** a distinct, explicit pair type carried alongside the DTO, never
inside it, from the three AI extractors down to `createRecipeWithRefs`'s existing `cook`
argument. The type system enforces completeness: every consumer of the changed return types must
be updated or `pnpm typecheck` fails.

### D-27-W3-03 — `recipeInstructions` accepts a step OBJECT **or** a plain string (object first)
Making the step object mandatory means a model that ignores the new shape produces a schema
violation, `Output.object` rejects, and **the import fails outright**. On a self-hosted fork
where the provider and model are user-configurable, that is a catastrophic regression for the
sake of a nice-to-have. **Resolution:**
`z.array(z.union([<step object>, z.string()]))`, object first, both branches `.describe()`d, plus
`coerceExtractionSteps()` in the normalizer that maps either branch onto the internal shape
(`{ text, ingredients: [], timers: [] }` for the string branch). A string-only response imports
exactly as it does today and simply earns no `.cook`. **`validateExtractionOutput` must keep
counting steps the same way** for both branches.

### D-27-W3-04 — THE COVERAGE GATE: no `.cook` unless every native ingredient is referenced
This is the decision that keeps W3 from silently deleting user data. With a `cook` argument,
`deriveProjectionTx` OWNS `recipe_ingredients` — the projection is built from the **per-step
refs**, not from the flat `recipeIngredient` list. If the model links 8 of 11 ingredients, the
imported recipe silently loses 3 ingredients from its ingredient list *and* from everything
the shopping list can add. **Resolution:** `buildCookFromExtraction` compares the
`normalizeIngredientLinkName`-normalized set of the native-system flat `recipeIngredient` list
against the union of the per-step ref names. If the ref set does not COVER the flat set, it
returns `null` — no `.cook`, legacy projection, import succeeds — and logs
`reason: "incomplete-ingredient-coverage"` with `missingCount` / `flatCount` (counts only, never
names or prose: an ingredient name is per-cookbook data, T-27-05). W5's backfill revisits every
`cook_source IS NULL` recipe anyway, so refusing costs nothing and losing an ingredient costs a
lot. Extra refs beyond the flat list are NOT a failure (the serializer already tolerates them).

### D-27-W3-05 — Opposite-system STEP rows are still written in the cook branch
`deriveProjectionTx` writes `recipe_ingredients` for both systems but `steps` for the **native
system only** (D-27-W2-05 — a converter cannot rewrite prose). W2's cook branch runs it
*instead of* the legacy step write, so on a create-with-`cook` the AI's already-extracted
opposite-system step prose would simply never be persisted, and the metric/US toggle would fall
back to an AI conversion call on every new import. That is a real, avoidable behaviour change
and it throws away output the user already paid for. **Resolution:** in the `cook` branch of
`createRecipeWithRefs` / `updateRecipeWithRefs`, the steps in `input.steps` whose `systemUsed`
is NOT the `.cook`'s native system are still written through the existing step writer, before
`deriveProjectionTx` runs. `deriveProjectionTx` does not read or write those rows, so there is
no conflict and no second owner. This is a ~5-line addition inside an existing branch, not a
re-plumb of the `cook` argument.

### D-27-W3-06 — An ordinary (no-`cook`) update of a recipe that HAS a `.cook` NULLs `cook_source`
W3 creates the first recipes with a `cook_source`. The recipe editor then saves through
`updateRecipeWithRefs` with **no** `cook` (the client has no linkage to send, and never will —
D-27-W2-01), which rewrites the projection from the client payload. The stored `.cook` is now
stale, yet `recipes.get` would keep shipping `cookTokens` derived from it and W4's renderer
would show the pre-edit steps. **Resolution:** the legacy branch of `updateRecipeWithRefs` sets
`cook_source = NULL` when the row currently has one. The recipe returns to the legacy path,
`cookTokens` goes `null`, and W5's backfill re-mints it. This preserves the invariant W4 and
W6's `0043` stand on: *a non-NULL `cook_source` describes the recipe it is attached to, and
parses cleanly.* Silent staleness is the only unacceptable outcome here.

### D-27-W3-07 — DUAL-system extraction is KEPT; "single-system extraction" is DEFERRED to W5
§4 says "assess enabling that in W3" — this is that assessment, and the answer is **not yet**.
W0's unit vocabulary still lacks `kilogram`, `fl oz` and `pint` (W2-SUMMARY, "still open"), so
`convertToSystem` collapses metric mass to `gram` and never produces fluid ounces. Deriving the
US list from the metric one today would ship visibly worse US output than the model already
gives us for free. **Resolution:** the model keeps emitting both systems; the `.cook` is minted
from the **native** system only (D-2), and the opposite system's ingredient rows come from
`deriveProjectionTx`'s deterministic conversion as W2 built it. Enabling single-system
extraction is a W5 item, after the unit vocabulary is filled. Task 4 must MEASURE the quality
delta (see its acceptance criteria) rather than assume it.

### D-27-W3-08 — Only the AI extraction paths become native in W3; JSON-LD / structured paste do NOT
"JSON-LD importer → per-step linkage" cannot mean what it literally says: JSON-LD gives a flat
step list and a flat ingredient list with **no linkage**, and the only way to invent linkage
from those is the heuristic name-matcher, which D-6/D-7 ban from every runtime path (it survives
only as one-time migration glue in W5). **Resolution:** the JSON-LD / python-scraper /
structured-paste / Mealie-archive paths produce **no** `cook` payload and keep writing the legacy
projection, exactly as today. What W3 actually delivers for the JSON-LD path is that
`normalizeRecipeFromJson`'s output flows through the same `ExtractedRecipe` channel with
`cook: null` — one code path, one shape, no special case. Those recipes are W5 backfill
territory. The alternative — running the banned heuristic at import time — is explicitly refused.

### D-27-W3-09 — W3 does NOT compute or store `cook_confidence`, and does NOT set `cook_review_needed`
`serializeWithReport`'s `LinkOutcome[]` is available and tempting. The confidence gate, its
threshold tuning against a restored dump, and the review queue are **W5** (§7). Setting
`cook_review_needed` for a refused mint would also pollute W5's queue with rows W5 already finds
via `cook_source IS NULL`. **Resolution:** both columns are untouched by W3. The signal W5 needs
is the error-level log `buildCookPayload` / `buildCookFromExtraction` already emit.

### D-27-W3-10 — W3 carries NO migration
`0041` (live) already provides `cook_source`. W3 only writes into it. `packages/db/src/migrations/`
and `meta/_journal.json` are untouched; the DB stays at migration **42**; the planned `0042`
backfill (W5) and `0043` NOT-NULL (W6) sequence is unchanged. Do NOT run `pnpm db:generate` /
`db:push` / `migrate` (D-27-W2-08 and CLAUDE.md).
</wave_decisions>

## What in W3 could break existing, user-visible behaviour — and how the executor must PROVE it did not

**R1 — RECIPE IMPORT stops working entirely because the model will not emit the new step shape.**
The single largest risk in this wave. `Output.object` validates against `recipeExtractionSchema`;
a mandatory step object turns every non-compliant model into a total import failure, and this
fork's provider/model are user-configurable. *Proof required:* D-27-W3-03's union, and Task 2's
assertions that an all-strings, an all-objects and a **mixed** `recipeInstructions` all parse,
all yield identical `validateExtractionOutput` counts, and that the all-strings DTO is deep-equal
to the pre-task output. If the union cannot be expressed for a provider's structured-output mode,
**stop and surface it** — do not make the object branch mandatory.

**R2 — RECIPE IMPORT silently loses ingredients.** With a `cook` argument,
`deriveProjectionTx` OWNS `recipe_ingredients` and builds them from the per-step refs, not from
the flat list. A model that links 8 of 11 ingredients would delete 3 from the recipe and from
everything the shopping list can add. *Proof required:* the coverage gate (D-27-W3-04) plus
Task 4's 8-of-11 test asserting 11 rows and `cook_source IS NULL`, and its 11-of-11 sibling.

**R3 — THE RECIPE LIST regresses now that `cook_source` is non-NULL.** W2 `.omit`ed the three
columns from `RecipeDashboardSchema` while nothing populated them; W3 is the first time the
omission is load-bearing. *Proof required:* Task 5 case 5 asserts the list/dashboard DTO for a
recipe that HAS a `cook_source` exposes no `cook*` key, and the `imported` realtime payload
likewise.

**R4 — THE SHOPPING LIST loses its "from recipe X" links.** `groceries.recipe_ingredient_id` is
`ON DELETE SET NULL`, and W3 is the first wave where `deriveProjectionTx` writes on a real user
action. *Proof required:* Task 4's FK test — import with a `.cook`, attach a grocery row,
re-derive with an updated `cook`, assert the SAME `recipe_ingredients.id` and a non-null FK.

**R5 — RECIPE UPDATE leaves a stale `.cook` behind.** The editor saves without linkage, so the

pre-edit steps. *Proof required:* D-27-W3-06 and its test (import → update → `cook_source IS
NULL`, `cookTokens === null`).

**R6 — THE METRIC/US TOGGLE degrades on every new import.** `deriveProjectionTx` writes native
